#+title: How ogent Works
#+options: toc:3 ^:nil

This is the architecture document: a mental model, one real request traced
end to end, a catalog of every module under =lisp/=, the invariants the
code enforces (each linked to the enforcing line), realistic failure
modes, and a demystification of the macro/advice/hook layers. Terms are
defined in the [[file:glossary.org][glossary]]; build/run commands live in [[file:../run.org][run.org]].

* Mental model

ogent is *a newsroom that publishes everything it does as Org text*.

Think of a newspaper office. Reporters don't wander in with rumors; every
story is filed with its sources stapled to it. That is the *context
engine*: a prompt never travels alone — the bibliographer
([[file:../lisp/ogent-context.el::864][ogent-context-build]]) chases each =@handle= citation, staples in pinned
exhibits, and prepends a table of contents listing every source with its
fingerprint ([[file:../lisp/ogent-context.el::480][the manifest]]). The editor-in-chief is the *dispatcher*
([[file:../lisp/ui/ogent-ui.el::1121][ogent-prompt-dispatch]]): it assigns the story to a wire service (a
gptel backend looked up in the [[file:../lisp/ogent-models.el::24][model registry]]) and opens a numbered
ticket (an [[file:../lisp/ui/ogent-ui.el::388][ogent-ui-request]]) with a reserved page in the Org notebook
where the copy streams in.

When the wire service wants to *do* something rather than say something —
read a file, run a shell command — it must pass the customs desk. Every
tool ships with a hazard label (its [[file:../lisp/ogent-tool-effects.el::15][effects]]: kind, target, scope, risk)
and one officer ([[file:../lisp/ogent-tool-effects.el::112][ogent-tool-effects-approval-required-p]]) decides who is
waved through: =read= passes, everything else needs a human signature.
Every execution is stamped into the tamper-evident logbook (the
[[file:../lisp/ogent-ledger.el::110][proof ledger]]) and the flight recorder ([[file:../lisp/ogent-analytics.el::196][analytics]]).

Downstairs, the *Armory* is the syndication department: a directory of
Org files that is itself a tiny operating system — agent personas, cron
jobs, conversations, skills — where work is done not by gptel but by
external coding CLIs (codex, claude, gemini, ...) booked through a travel
agent ([[file:../lisp/ogent-armory-runner.el::426][the runner's plan]]) and filed back as Org
[[file:../lisp/ogent-armory-conversations.el::268][conversation records]]. Agents may propose new work, but their proposals
sit in an outbox ([[file:../lisp/ogent-armory-actions.el::489][actions]]) until a human stamps them.

The load-bearing idea throughout: *Org text is the database, the UI, and
the audit log at once*. Requests are headlines, tool calls are drawers,
armory state is property drawers, sessions are files. If you can read an
Org file, you can read everything ogent ever did.

* The walkthrough
:PROPERTIES:
:CUSTOM_ID: the-walkthrough
:END:

One real interaction, end to end. The setup: you are visiting
=lisp/ogent-tool-effects.el= in this repo, =ogent-mode= is on, gptel is
authenticated against Anthropic via OAuth, and the model
=claude-sonnet-4-6= is selected. You press =C-c . p=, type

#+begin_example
Which files reference ogent-tool-spec-confirm-p? Use grep, then run the
test suite to confirm the tree is green.
#+end_example

and hit =RET=. Here is everything that happens, with the actual values.

** 1. The keypress

=C-c . p= lives in [[file:../lisp/ogent-core.el::182][ogent-mode-map]], which was generated at load time
from the declarative [[file:../lisp/ogent-keys.el::110][action registry]] by
[[file:../lisp/ogent-keys.el::397][ogent-setup-all-bindings]] — the same table that produces the Evil
(=SPC o p=) and Doom bindings. The =prompt= action's command is
[[file:../lisp/ui/ogent-ui.el::1121][ogent-prompt-dispatch]], a transient prefix like magit's dispatch. Its
infixes stash your choices in defvars: the provider infix sets
=gptel-backend= / =gptel-model=, and the prompt infix holds your text.
Pressing =RET= fires [[file:../lisp/ui/ogent-ui.el::1109][ogent--suffix-send-action]], which flashes the
mode-line via [[file:../lisp/ui/ogent-ui-theme.el::317][ogent-theme-flash]] and calls
[[file:../lisp/ui/ogent-ui.el::565][ogent--send-with-current-model]].

** 2. Companion and context

Your source buffer is elisp, not Org, so
[[file:../lisp/ui/ogent-ui.el::610][ogent-ui--ensure-companion-context]] asks
[[file:../lisp/ogent-companion.el::98][ogent-companion-get-or-create]] for the Org sidecar — a buffer named
=*ogent:ogent-tool-effects.el*= is created, linked both ways, and recorded
in the persistent link registry so the pairing survives restarts.

Inside the companion, [[file:../lisp/ogent-context.el::931][ogent-context-build-with-source]] builds the
context plist: an [[file:../lisp/ogent-context.el::346][ogent-source-context]] snapshot of
=ogent-tool-effects.el= (mode =emacs-lisp-mode=, full buffer content), any
=@handles= found (none in this prompt), and the global
[[file:../lisp/ogent-context.el::87][pinned items]] (also none). The pre-send gate
[[file:../lisp/ogent-core.el::112][ogent-validate-and-prompt]] checks for unresolved handles — with
=ogent-validate-before-send= at its default =nil= it passes silently.

** 3. The request ticket

[[file:../lisp/ui/ogent-ui.el::697][ogent-response-function]] (default
[[file:../lisp/ui/ogent-ui.el::1512][ogent-ui-prepare-response-block]]) mints the request. First
[[file:../lisp/ui/ogent-ui.el::1436][ogent-ui--create-response-block]] writes the Org skeleton into the
companion:

#+begin_example
** Request: Which files reference ogent-tool-spec-confirm-p? Use grep, ...
:PROPERTIES:
:OGENT_PROMPT: Which files reference ogent-tool-spec-confirm-p? Use grep, then summarize.
:END:
#+begin_src text :model claude-sonnet-4-6 :backend Claude :status waiting
<rendered payload — folded immediately>
#+end_src

*** Response (claude-sonnet-4-6)
#+end_example

A struct =ogent-ui-request= is created with =:id "ogent-request-1"= (ids
come from the counter in [[file:../lisp/ui/ogent-ui.el::1306][ogent-ui--next-request-id]]), a streaming
marker parked after the Response headline, and =:status 'wait=.
[[file:../lisp/ui/ogent-ui.el::1494][ogent-ui-register-request]] puts it in
[[file:../lisp/ui/ogent-ui.el::708][ogent-ui--request-table]], stamps the start into the proof ledger
([[file:../lisp/ogent-ledger.el::135][ogent-ledger-record-request-start]] — hash of the prompt, not the
text), starts the analytics clock
([[file:../lisp/ogent-analytics.el::186][ogent-analytics-start-request]]), and creates the animated left-margin
indicator ([[file:../lisp/ui/ogent-ui-status.el::250][ogent-status-set-request]]).

** 4. The provider call

[[file:../lisp/ui/ogent-ui.el::2240][ogent-ui--send-request]] renders the final payload with
[[file:../lisp/ogent-context.el::625][ogent-context-render-prompt]] — manifest first:

#+begin_example
Context manifest:
- source: ogent-tool-effects.el (emacs-lisp-mode, 4.7k chars, sha256:0f3a…)
#+end_example

then binds the enabled tool objects from
[[file:../lisp/ogent-models.el::305][ogent-tools-enabled-list]] into =gptel-tools= and calls
=gptel-request= with the callback from
[[file:../lisp/ui/ogent-ui.el::1850][ogent-ui--make-callback]] closed over ="ogent-request-1"=.

When =ogent-multi-turn-history= is non-nil (the default), the PROMPT
argument is not a single string: completed =** Request:= /
=*** Response= pairs earlier in the buffer are replayed as alternating
user/assistant turns. The raw user prompt of each turn is read back
from the =OGENT_PROMPT= property the skeleton writer persisted
(newline-encoded; [[file:../lisp/ui/ogent-ui.el::1342][ogent-ui--encode-prompt-property]]), the
assistant turn from the matching Response body
([[file:../lisp/ui/ogent-ui.el::1382][ogent-ui--conversation-history]]; fan-out picks the same model's
response, empty/aborted turns are skipped). The history is then
compacted to =ogent-multi-turn-token-budget= estimated tokens by
evicting the oldest pairs first
([[file:../lisp/ui/ogent-ui.el::1421][ogent-ui--compact-history]]) and sent as =(user-1 reply-1 … prompt)=.
The request also binds =gptel-cache= from =ogent-gptel-cache= and
copies any registry =:request-params= / =:capabilities= onto the gptel
model symbol ([[file:../lisp/ogent-models.el::86][ogent-models-apply-gptel-props]]) — on cache-capable
Anthropic models the replayed prefix becomes the cached prompt prefix
across turns.

Because you authenticated with Anthropic OAuth, two pieces of advice fire
inside gptel: [[file:../lisp/ogent-anthropic-oauth.el::578][ogent-anthropic-oauth--request-data-advice]] swaps the
system message to the exact Claude Code prefix (OAuth bearer tokens are
rejected without it), and
[[file:../lisp/ogent-anthropic-oauth.el::602][ogent-anthropic-oauth--curl-args-advice]] injects =Authorization:
Bearer …= plus the =anthropic-beta= headers — refreshing the token first
if it is near expiry ([[file:../lisp/ogent-anthropic-oauth.el::329][ogent-anthropic-oauth--ensure-valid-token]]).

** 5. Streaming

Chunks arrive in the callback and flow through
[[file:../lisp/ui/ogent-ui.el::1650][ogent-ui--append-response]]: inserted at the request's marker, model
headings demoted below the =*** Response= level, window scrolled if you
haven't moved away. The first chunk flips status to ='typing= via
[[file:../lisp/ui/ogent-ui.el::1698][ogent-ui--update-status]] (margin lamp starts spinning,
[[file:../lisp/ui/ogent-ui-status.el::217][ogent-status--start-animation]]) and stamps the time-to-first-token
([[file:../lisp/ogent-analytics.el::191][ogent-analytics-first-token]]).

** 6. Tool call #1 — grep, auto-approved

The model answers with a tool call:
=grep(pattern: "ogent-tool-spec-confirm-p", glob_filter: "*.el")=. The
callback routes it to [[file:../lisp/ui/ogent-ui.el::1910][ogent-ui--handle-tool-calls]], which first asks
[[file:../lisp/ogent-tool-approval.el::162][ogent-tool-approval-check]]. The decision cascade: approval is
enabled; =grep= is not session-denied; not in your
[[file:../lisp/ogent-tool-approval.el::15][allow-list]]; so the spec policy decides. =grep='s spec in
[[file:../lisp/ogent-tools.el::750][ogent-tools-default-registry]] declares =:effects ((:kind read :target
filesystem :scope workspace :risk low))= — =read= is not in
[[file:../lisp/ogent-tool-effects.el::19][ogent-tool-effects-approval-kinds]] and =low= is below the =high=
threshold, so [[file:../lisp/ogent-tool-effects.el::112][ogent-tool-effects-approval-required-p]] says no
approval needed: *approved, no prompt shown*.

[[file:../lisp/ui/ogent-ui.el::1977][ogent-ui--execute-tool]] records
[[file:../lisp/ogent-ledger.el::162][ogent-ledger-record-tool-start]] (with those effects), runs
[[file:../lisp/ogent-tools.el::291][ogent-tool--grep]] (ripgrep under the hood, project root resolved by
[[file:../lisp/ogent-tools.el::170][ogent-tools--project-root]]), records the finish with a result hash,
and renders the result into the transcript as a collapsible
[[file:../lisp/ui/ogent-ui.el::2249][tool drawer]]:

#+begin_example
:TOOL_CALL (grep):
pattern: ogent-tool-spec-confirm-p
lisp/ogent-models.el:251:(defun ogent-tool-spec-confirm-p (spec)
lisp/ogent-models.el:284:  :confirm (and (ogent-tool-spec-confirm-p spec) t)
test/ogent-models-tests.el:…
:END:
#+end_example

The result is fed back to the model through gptel's tool loop.

** 7. Tool call #2 — bash, human gate

Next the model calls =bash(command: "make test")=. Same cascade — but the
bash spec is double-flagged: =:confirm t= *and* =:effects ((:kind execute
:target shell :scope unrestricted :risk critical) …)=. Either alone forces
the prompt. [[file:../lisp/ogent-tool-approval.el::122][ogent-tool--prompt-approval]] takes over the echo area:

#+begin_example
Tool request: bash
  command: make test
Effects: execute shell (unrestricted) [critical]
Execute? (y)es (n)o (a)lways (e)dit:
#+end_example

You press =y=. (Had you pressed =a=, the pattern =bash(command:make *)= —
narrowed to the command's first word — would be persisted into
[[file:../lisp/ogent-tool-approval.el::15][ogent-tool-allow-list]] via =customize-save-variable=.) Because
=ogent-stream-tool-output= defaults to =t=, execution goes through the
async path [[file:../lisp/ui/ogent-ui.el::1850][ogent-ui--execute-tool-async]] →
[[file:../lisp/ogent-tools.el::577][ogent-tool--bash-async]]: a =make-process= with a kill-timer at
[[file:../lisp/ogent-tools.el::31][ogent-tools-shell-timeout]] (120s), stdout streaming into a live
[[file:../lisp/ui/ogent-ui.el::2301][streaming drawer]] in the transcript while the spinner shows byte/line
counts. =make test= exits 0; the drawer is finalized with the exit code
and the (truncated, per [[file:../lisp/ogent-tools.el::26][ogent-tools-max-output-chars]]) output goes
back to the model.

** 8. The answer and the close

The model streams its final prose ("=ogent-tool-spec-confirm-p= is
referenced from the gptel tool-registration bridge … the suite passes").
gptel signals completion; the callback (or, if the stream ends
ambiguously, the fallback
[[file:../lisp/ui/ogent-ui.el::1755][ogent-ui--gptel-post-response-handler]]) calls
[[file:../lisp/ui/ogent-ui.el::1596][ogent-ui--close-response]] for ="ogent-request-1"=: status → ='done=,
the =:status waiting= cookie in the prompt block is rewritten, the prompt
block stays folded, a green flash fires, and three ledgers settle up —
[[file:../lisp/ogent-analytics.el::196][ogent-analytics-record-completion]] INSERTs a row (model
="claude-sonnet-4-6"=, estimated tokens, TTFT, latency) into
=.ogent-analytics.db=; [[file:../lisp/ogent-ledger.el::147][ogent-ledger-record-request-finish]] appends the
hashed completion event; and [[file:../lisp/ogent-core.el::290][ogent-notify-completion]] runs
=ogent-after-completion-hook=, which lets the
[[file:../lisp/ogent-completions.el::491][completions review workflow]] notice the new Response sibling. The
margin lamp turns into a checkmark and is cleared two seconds later.

Total state touched: one Org subtree in the companion buffer, one row in
the SQLite analytics DB, ledger events (if enabled), the request moved
from the active table into [[file:../lisp/ui/ogent-ui.el::1576][retry history]].

** Coda: the same shape in the Armory

The Armory pipeline rhymes, with external CLIs instead of gptel:
[[file:../lisp/ogent-armory-compose.el::263][ogent-armory-compose]] expands =@mentions= (instead of @handles),
[[file:../lisp/ogent-armory-runner.el::426][ogent-armory-runner-plan]] resolves agent/provider/model (instead of
the model registry), [[file:../lisp/ogent-armory-runner.el::799][ogent-armory-runner-start]] spawns =codex exec=
(instead of =gptel-request=), and instead of a response block the run is
persisted as a conversation directory — created at start by
[[file:../lisp/ogent-armory-runner.el::663][ogent-armory-runner--create-conversation]] and finalized by
[[file:../lisp/ogent-armory-runner.el::754][ogent-armory-runner--finalize-conversation]], which parses the agent's
structured footer and files any proposed actions behind the
[[file:../lisp/ogent-armory-actions.el::489][human approval gate]].

* Module catalog

Every file under =lisp/= and =lisp/ui/=, grouped by subsystem. Each entry
gives the module's Role, Design, an Intuition, a Trace of the typical call
path, and exact-line links to its major defuns, defcustoms, and state.

** Entry point and core

*** [[file:../lisp/ogent.el::1][ogent.el]] — the bootstrap
:PROPERTIES:
:CUSTOM_ID: mod-ogent
:END:

Role: Umbrella entry point — fixes up the =load-path=, hardens against
stale Org byte-code, =require=s every module in dependency order, and runs
one-time install steps (default tools, the =ogent= Babel language).

Design: Almost no logic of its own. A top-level =let*= at
[[file:../lisp/ogent.el::52][lisp/ogent.el:52]] pushes the entry file's directory, its =ui/= subdir,
and their truename variants onto =load-path= (Doom/straight symlink builds
otherwise miss sibling files), then a flat cascade of ~37 =require=s
([[file:../lisp/ogent.el::101][lisp/ogent.el:101]]) defines the canonical load order. A small shim
around =org-capture-templates-contexts= protects against old compiled Org.

Intuition: The table of contents plus the power switch — it does nothing
itself; it makes sure everything else is present and turned on in order.

Trace: =(require 'ogent)= → load-path bootstrap → defensive
[[file:../lisp/ogent.el::68][org-capture-templates-contexts]] defvar +
[[file:../lisp/ogent.el::71][ogent--ensure-org-capture-templates-contexts]] → require cascade
(context → models → tools → core → UI → armory → presets → analytics) →
[[file:../lisp/ogent.el::141][ogent-tools-install-defaults call]] populates the tool registry →
=with-eval-after-load 'org= registers =(ogent . t)= in
=org-babel-load-languages= ([[file:../lisp/ogent.el::144][lisp/ogent.el:144]]).

Key definitions: [[file:../lisp/ogent.el::79][ogent--with-org-capture-templates-contexts]] (the
:around advice body) and [[file:../lisp/ogent.el::88][ogent--advise-org-capture-contexts]]
(idempotent installer) — applied to =org-capture= and friends inside
=with-eval-after-load 'org-capture=.

*** [[file:../lisp/ogent-core.el::1][ogent-core.el]] — minor mode, validation, notification
:PROPERTIES:
:CUSTOM_ID: mod-ogent-core
:END:

Role: Defines =ogent-mode= / =ogent-global-mode= and their keymap,
pre-send validation of @handles and prompt requirements, completion
notification, and the lightweight standalone commands (=ogent-ask=,
=ogent-open-block=, Question→Response session prompting).

Design: Four loosely coupled sections — validation utilities that read
=:dependencies= off the context plist and write =:validation-warnings=
back; the minor-mode scaffold whose keymap is generated from the
ogent-keys action registry; a notification layer driven by hooks and a
request plist; and gptel streaming commands that either accumulate text in
module defvars (=ogent-ask=) or stream into an Org =Response= headline via
a marker (sessions). =ogent-global-mode= hand-wraps the internal
globalized mode so enabling it never disturbs window layout.

Intuition: The front door and the doorman — it turns ogent on in buffers,
checks your papers before requests leave, and rings the bell when
responses come back.

Trace: ogent-ui calls [[file:../lisp/ogent-core.el::112][ogent-validate-and-prompt]] →
[[file:../lisp/ogent-core.el::91][ogent-context-validate]] collects missing handles → pcase on
[[file:../lisp/ogent-core.el::64][ogent-validate-before-send]] decides proceed/warn/confirm. On
completion, [[file:../lisp/ogent-core.el::290][ogent-notify-completion]] runs
[[file:../lisp/ogent-core.el::84][ogent-after-completion-hook]] then =message= or
[[file:../lisp/ogent-core.el::263][ogent--flash-modeline]]. For inline sessions, =C-c C-c= →
[[file:../lisp/ogent-core.el::615][ogent-session--ctrl-c-ctrl-c-handler]] →
[[file:../lisp/ogent-core.el::539][ogent-session-prompt-from-question]] →
[[file:../lisp/ogent-core.el::502][ogent-session--create-response-headline]] → =gptel-request= with the
marker-based [[file:../lisp/ogent-core.el::584][ogent-session--stream-callback]].

Key definitions: defcustoms [[file:../lisp/ogent-core.el::58][ogent-after-request-hook]],
[[file:../lisp/ogent-core.el::64][ogent-validate-before-send]] (default nil),
[[file:../lisp/ogent-core.el::74][ogent-notify-on-completion]] (nil/'message/'modeline-flash),
[[file:../lisp/ogent-core.el::84][ogent-after-completion-hook]],
[[file:../lisp/ogent-core.el::321][ogent-ask-display-function]]. Modes:
[[file:../lisp/ogent-core.el::192][ogent-mode]] (wires capf + C-c C-c hook in Org buffers),
[[file:../lisp/ogent-core.el::229][ogent-global-mode--internal]] and its wrapper
[[file:../lisp/ogent-core.el::233][ogent-global-mode]]. Commands: [[file:../lisp/ogent-core.el::388][ogent-ask]] (popup Q&A; callback
factory [[file:../lisp/ogent-core.el::359][ogent-ask--make-callback]]),
[[file:../lisp/ogent-core.el::420][ogent-open-block]]. State: [[file:../lisp/ogent-core.el::182][ogent-mode-map]] (built at defvar time),

*** [[file:../lisp/ogent-keys.el::1][ogent-keys.el]] — one registry, every binding frontend
:PROPERTIES:
:CUSTOM_ID: mod-ogent-keys
:END:

Role: The unified keybinding system: a declarative action registry
generates vanilla (=C-c .=), Evil leader (=SPC o=), Doom leader, review
(=C-c ,=), and which-key bindings; also home of the canonical Evil
display-buffer integration used by every read-only ogent buffer.

Design: [[file:../lisp/ogent-keys.el::110][ogent-action-registry]] is a single source of truth —

* Invariants

The rules the code actively enforces, each with the enforcing line.

1. *No tool with non-read effects auto-executes.* The policy predicate
   [[file:../lisp/ogent-tool-effects.el::112][ogent-tool-effects-approval-required-p]] requires approval when any
   effect kind is in [[file:../lisp/ogent-tool-effects.el::19][ogent-tool-effects-approval-kinds]] (=write
   execute network git issue emacs-state= — everything but =read=) *or*
   combined risk reaches =high= on the rank scale at
   [[file:../lisp/ogent-tool-effects.el::23][ogent-tool-effects--risk-ranks]].

2. *The approval gate is enforced in both execution paths.* ogent's own
   pipeline checks the spec policy inside
   [[file:../lisp/ogent-tool-approval.el::162][ogent-tool-approval-check]] — a tool auto-runs only if it has a
   spec AND neither =:confirm= nor approval-requiring effects; everything else
   falls to the interactive prompt
   [[file:../lisp/ogent-tool-approval.el::122][ogent-tool--prompt-approval]]. gptel-native execution is bridged by
   [[file:../lisp/ogent-models.el::251][ogent-tool-spec-confirm-p]], passed explicitly as =:confirm= at
   registration ([[file:../lisp/ogent-models.el::284][lisp/ogent-models.el:284]]) so gptel's own runner
   prompts under the same policy.

3. *MCP tools always require approval by default.* Every tool discovered
   from an MCP server is stamped with
   [[file:../lisp/ogent-mcp.el::65][ogent-mcp-default-tool-effects]] — =(:kind network :target external
   :scope unrestricted :risk high)= — inside
   [[file:../lisp/ogent-mcp.el::325][ogent-mcp--register-tools]] (the =copy-tree= there also prevents
   registered specs from aliasing the defcustom). Both clauses of
   invariant 1 fire: =network= is an approval kind AND =high= trips the
   risk threshold. Before this rule (commit =d6421db=), an effects-less
   MCP spec satisfied the auto-run clause and could execute silently.

4. *Malformed tool calls are denied, not guessed.* An unresolvable tool
   name short-circuits to ='denied= at the top of the gate
   ([[file:../lisp/ui/ogent-ui.el::1937][lisp/ui/ogent-ui.el:1937]]).

5. *Dangerous built-ins are double-flagged.* =bash=, =write-file=, and
   =edit-file= carry both =:confirm t= and high/critical effects in
   [[file:../lisp/ogent-tools.el::750][ogent-tools-default-registry]] — either alone forces approval, so a
   regression in one layer doesn't open the gate.

6. *"Always allow" narrows shell patterns to the command word.* When you
   answer =a= to a bash prompt, the persisted allow-list pattern is
   narrowed to the command's first word in
   [[file:../lisp/ui/ogent-ui.el::245][ogent-tool--add-to-allow-list]] — approving =make test= does not
   approve arbitrary commands.

7. *Every tool execution is auditable.* The executor records start and
   finish (with declared effects and a result hash) to the proof ledger
   in [[file:../lisp/ui/ogent-ui.el::1977][ogent-ui--execute-tool]] via
   [[file:../lisp/ogent-ledger.el::162][ogent-ledger-record-tool-start]] /
   [[file:../lisp/ogent-ledger.el::170][ogent-ledger-record-tool-finish]]; the ledger only ever appends
   ([[file:../lisp/ogent-ledger.el::88][ogent-ledger--append-event]] uses =append-to-file=).

8. *Edit-tool writes go through review, not straight to disk.* When the
   model calls write-file/edit-file, the UI generates a pending diff
   ([[file:../lisp/ui/ogent-ui.el::2636][ogent-ui--generate-diff]]) that must be accepted via
   [[file:../lisp/ui/ogent-ui.el::2777][ogent-diff-accept]] before the file changes.

9. *=.ogent.el= is parsed, never evaluated.* Project preset files go
   through =read= plus a variable whitelist in
   [[file:../lisp/ogent-presets.el::149][ogent-presets--safe-read]] against
   [[file:../lisp/ogent-presets.el::138][ogent-presets--allowed-variables]] — a hostile repo cannot execute
   code via its settings file.

10. *An edit applies only if its old text matches exactly once.*
    [[file:../lisp/ogent-edit-parse.el::66][ogent-edit-validate]] requires a unique literal match in the source
    buffer; zero or multiple matches mark the edit ='error= and it is
    never auto-applied (filter at
    [[file:../lisp/ogent-edit-parse.el::114][ogent-edit-filter-valid]]).

11. *Batch edits apply in reverse position order* so earlier
    replacements cannot shift later positions —
    [[file:../lisp/ogent-edit.el::705][ogent-edit-auto-apply-all]] and
    [[file:../lisp/ogent-edit-display.el::107][ogent-edit-apply-all-as-smerge]] both sort descending.

12. *Armory agents cannot launch work; humans dispatch proposals.*
    Parsed actions are stored with status ="proposed"= and only
    =:valid= + ="approved"= actions pass the gate in
    [[file:../lisp/ogent-armory-actions.el::489][ogent-armory-actions-dispatch]] (validity check at
    [[file:../lisp/ogent-armory-actions.el::250][ogent-armory-actions-validate]] — nonexistent target agents can
    never dispatch).

13. *Model/preset/tool identity flows through registries.* Unknown model
    ids fail loudly at [[file:../lisp/ogent-models.el::69][ogent-models-ensure]] (user-error) rather than
    falling back silently.

14. *Linked git worktrees share one issue database.* Before armory runs
    and br calls, [[file:../lisp/ogent-issues-bd.el::128][ogent-issues-bd-ensure-worktree-redirect]] writes
    =.beads/redirect= in linked worktrees so parallel agents do not fork
    the tracker (gated by
    [[file:../lisp/ogent-armory-runner.el::143][ogent-armory-runner-ensure-beads-redirect]], default t).

15. *Anthropic OAuth requests always carry the Claude Code identity.*
    [[file:../lisp/ogent-anthropic-oauth.el::578][ogent-anthropic-oauth--request-data-advice]] binds the exact system
    prefix unconditionally when OAuth is active — without it the
    provider rejects bearer-token requests.

What is *not* enforced, deliberately: there is no OS-level sandbox. The
effects =:scope= (=workspace= / =unrestricted=) is declarative risk
metadata for the approval prompt;
[[file:../lisp/ogent-tools.el::180][ogent-tools--resolve-path]] anchors relative paths to the project root
but absolute paths pass through. The human approval gate *is* the
sandbox.

* Failure modes

What realistically breaks, and where it is detected.

- *Stale =.elc= vs macro-expansion-time decisions.* Three modes pick
  their parent class at byte-compile time: the issues mode macro
  ([[file:../lisp/ogent-issues.el::528][ogent-issues--define-mode]]), the armory status mode macro
  ([[file:../lisp/ogent-armory-status.el::352][ogent-armory-status--define-mode]]), and the armory UI section
  modes ([[file:../lisp/ui/ogent-ui-armory.el::752][ogent-armory-ui--define-section-mode]]). Compile without
  magit-section but run with it (or vice versa) and sections silently
  degrade or break; runtime capability checks
  ([[file:../lisp/ogent-armory-status.el::324][ogent-armory-status--magit-section-usable-p]]) soften the blow.
  The same trap hits the Evil display setup (mirroring never runs,
  printed key hints "do nothing"). Cure: =make recompile= (see
  [[file:../run.org][run.org]]).

- *gptel callback shape drift.* Stream-completion detection is
  heuristic. If gptel changes its info-plist shape, requests stick at
  "typing" — the fallback
  [[file:../lisp/ui/ogent-ui.el::1755][ogent-ui--gptel-post-response-handler]] closes any unclosed request
  whose handle matches the response start. Same heuristics exist in
  [[file:../lisp/ogent-edit.el::107][ogent-edit--make-callback]] and the
  [[file:../lisp/ogent-codemap.el::840][codemap generator]] (a stuck entry in its pending table then reports
  "Request already pending" on retry).

- *Duplicate tool execution across stream chunks.* gptel reuses one
  =:tool-use= list across chunks; the identity guard on the request's
  =handled-tool-use= slot ([[file:../lisp/ui/ogent-ui.el::1720][lisp/ui/ogent-ui.el:1720]]) is what
  prevents tools firing once per chunk. A regression here re-runs tools.

- *Provider/auth failures.* Access-class errors (401, model_not_found,
  quota, billing) are classified by
  [[file:../lisp/ogent-provider-fallback.el::60][ogent-provider-access-error-p]] and surface the login offer; OAuth
  refresh failure clears tokens and asks for re-login
  ([[file:../lisp/ogent-anthropic-oauth.el::329][ogent-anthropic-oauth--ensure-valid-token]]). gptel struct-layout
  changes silently break the OAuth header injection (raw =aset= by slot
  offset at [[file:../lisp/ogent-anthropic-oauth.el::602][ogent-anthropic-oauth--curl-args-advice]]) — debug with
  =ogent-anthropic-oauth-debug=.

- *Edits applied against a changed buffer.* Validation pins positions at
  parse time; the magit-style review buffer applies without
  re-validating ([[file:../lisp/ogent-edit-diff.el::462][ogent-edit-diff--apply-edit]]), so editing the
  source between display and accept can replace the wrong region. The
  smerge path is safer (markers are physical). Repeated edit IDs across
  responses (counter reset at
  [[file:../lisp/ogent-edit-format.el::65][ogent-edit--reset-counter]]) can stamp the wrong companion log
  entry.

- *Model ignores the wire format.* In the edit pipeline, malformed
  SEARCH/REPLACE markers parse to zero edits (scan loop at
  [[file:../lisp/ogent-edit-parse.el::14][ogent-edit-parse-response]]) — visible only as "0 edit blocks". In
  the Armory, an agent that ignores the response contract is detected
  downstream when [[file:../lisp/ogent-armory-conversations.el::831][ogent-armory-conversation-parse-output]] finds no
  machine footer; non-zero exits are triaged by
  [[file:../lisp/ogent-armory-adapter.el::297][ogent-armory-adapter-classify-error]].

- *Async tool processes hang or flood.* Shell timeouts kill the process
  and raise ([[file:../lisp/ogent-tools.el::486][ogent-tool--bash]] deadline; async variant uses a
  =run-at-time= killer). Output is clamped by
  [[file:../lisp/ogent-tools.el::187][ogent-tools--truncate-output]]. Leftovers are killable via
  [[file:../lisp/ogent-tools.el::674][ogent-tools-cancel-all]]. MCP servers that stop answering surface as
  synthesized =-32000= timeouts in
  [[file:../lisp/ogent-mcp.el::244][ogent-mcp--request-sync]]; dead server processes flip connection
  status via the sentinel
  ([[file:../lisp/ogent-mcp.el::176][ogent-mcp--make-process-sentinel]]) — but tools registered before a
  reconnect still proxy to the dead connection until re-registered
  (closure capture in [[file:../lisp/ogent-mcp.el::325][ogent-mcp--register-tools]]).

- *Handles that don't resolve.* A typo'd =@handle= becomes a
  =:missing-p= dependency reported in the rendered prompt's "Missing
  @handles" section ([[file:../lisp/ogent-context.el::625][ogent-context-render-prompt]]) and optionally
  gates sending via [[file:../lisp/ogent-core.el::64][ogent-validate-before-send]]. In babel blocks it
  is embedded as "(unresolved)" ([[file:../lisp/ob-ogent.el::79][ob-ogent--resolve-context]]) — the
  model silently answers without that context.

- *Armory records with malformed metadata.* Bad cron/boolean job
  properties are collected with friendly messages by
  [[file:../lisp/ogent-armory.el::1058][ogent-armory-job-validation-errors]]; a conversation index whose
  drawer lost its =:END:= raises in the drawer patcher
  ([[file:../lisp/ogent-armory-conversations.el::508][ogent-armory-conversations--update-property-drawer]]); a missing
  armory index is a =user-error= in
  [[file:../lisp/ogent-armory.el::409][ogent-armory-read-index]]. Malformed action JSON is silently
  dropped (=condition-case= → nil in
  [[file:../lisp/ogent-armory-actions.el::172][ogent-armory-actions--json-actions]]).

- *Hot-hook slowdowns.* [[file:../lisp/ogent-presets.el::394][ogent-presets-mode]] and the session
  auto-saver ride =buffer-list-update-hook= / =kill-buffer-hook=
  Emacs-wide; an error or slow filesystem inside them degrades all of
  Emacs. Both are guarded (file-buffer check + caches; mode/id checks).

- *SQLite-less Emacs.* Analytics degrades gracefully — connection getter
  returns nil ([[file:../lisp/ogent-analytics.el::80][ogent-analytics--get-db]]), dashboard prints "SQLite
  not available", exports raise =user-error=.

* No magic

The non-obvious layers, demystified.

** Load order

=(require 'ogent)= does four things in sequence
([[file:../lisp/ogent.el::52][lisp/ogent.el:52]] onward): fix =load-path= (including truenames, for
Doom/straight symlinked builds), shim old Org capture byte-code, require
~37 modules in a deliberate order (=ogent-ui-theme= before =ogent-ui=;
registries before their consumers; the armory stack bottom-up:
armory → evil → adapter → conversations → data → skills → settings →
runner → schedule → actions → compose → status), then install default
tools ([[file:../lisp/ogent.el::141][lisp/ogent.el:141]]) and register the Babel language. Three
load-time side effects worth knowing: the prompt registry populates when
=ogent-prompts= loads ([[file:../lisp/ogent-prompts.el::417][lisp/ogent-prompts.el:417]]), response tracking
hooks into gptel when =ogent-notes= loads ([[file:../lisp/ogent-notes.el::84][lisp/ogent-notes.el:84]]),
and the adapter registry resets when =ogent-armory-adapter= loads
([[file:../lisp/ogent-armory-adapter.el::537][lisp/ogent-armory-adapter.el:537]]).

** The macro layers

- *Capability-switched modes.* [[file:../lisp/ogent-issues.el::528][ogent-issues--define-mode]],
  [[file:../lisp/ogent-armory-status.el::352][ogent-armory-status--define-mode]], and
  [[file:../lisp/ui/ogent-ui-armory.el::752][ogent-armory-ui--define-section-mode]] choose
  =magit-section-mode= vs =special-mode= as the parent *when the macro
  expands* (an =eval-and-compile= probe). This is why magit-section
  availability must match between compile time and runtime — the
  stale-=.elc= trap described under Failure modes.

- *Section wrappers.* [[file:../lisp/ogent-armory-status.el::591][ogent-armory-status--with-section]] and
  [[file:../lisp/ui/ogent-ui-armory.el::712][ogent-armory-ui--with-section]] emit real magit sections when
  usable and plain text otherwise, so rendering code is written once.

- *Compile-away debug.* [[file:../lisp/ogent-debug.el::224][ogent-debug]] expands to =nil= unless
  =ogent-debug-enabled= was set at expansion time — enabling it later
  requires recompiling (the macro warns about this).

- *Deferred Evil setup.* [[file:../lisp/ogent-keys.el::521][ogent-evil-setup-display-mode]] expands to a
  =with-eval-after-load 'evil= block; nothing Evil-related runs unless
  Evil loads.

- *Smaller ones:* [[file:../lisp/ogent-context.el::976][ogent-context-with-lazy]] (thunk-deferred context
  builds), [[file:../lisp/ogent-armory-schedule.el::424][ogent-armory-with-agenda-files]] (dynamically shadows
  =org-agenda-files=), and the transient prefixes
  (=ogent-prompt-dispatch=, =ogent-edit-menu=, =ogent-navigate=,
  =ogent-issues-dispatch=, armory dispatches) which are
  =transient-define-prefix= macros, not functions.

** The advice

Exactly three subsystems use advice; everything else is hooks.

1. *Anthropic OAuth* (the heaviest): =:around=
   =gptel-curl--get-args= to inject bearer/beta headers (installed at
   [[file:../lisp/ogent-anthropic-oauth.el::625][lisp/ogent-anthropic-oauth.el:625]]) and =:around= the
   =gptel--request-data= generic to pin the Claude Code system prefix
   ([[file:../lisp/ogent-anthropic-oauth.el::633][lisp/ogent-anthropic-oauth.el:633]]). Both no-op when no OAuth
   tokens exist.
2. *Org capture shim*: =:around= =org-capture= and friends, purely
   defensive against stale Org builds
   ([[file:../lisp/ogent.el::88][ogent--advise-org-capture-contexts]]).
3. *Analytics outcomes*: =:around= the completions accept/reject
   commands to stamp outcome rows, installed by
   [[file:../lisp/ogent-analytics.el::565][ogent-analytics-setup]].

** The hooks

The request loop is wired by hooks, not polling:
=gptel-post-response-functions= gets
[[file:../lisp/ui/ogent-ui.el::1787][ogent-ui--register-gptel-hook]]'s fallback handler;
=gptel-post-response-hook= feeds [[file:../lisp/ogent-notes.el::55][notes tracking]];
=gptel-pre-request-hook= feeds [[file:../lisp/ogent-analytics.el::186][analytics timing]] and
[[file:../lisp/ogent-debug.el::88][debug logging]]. Buffer-locally, =ogent-mode= adds the @handle capf and
the =C-c C-c= session handler ([[file:../lisp/ogent-core.el::192][ogent-mode]] body); the UI adds a
=post-command-hook= for auto-scroll while streaming. Globally,
=kill-buffer-hook= carries session auto-save
([[file:../lisp/ogent-session.el::362][ogent-persist--maybe-auto-save]]), =buffer-list-update-hook= carries
project preset loading ([[file:../lisp/ogent-presets.el::220][ogent-presets--on-buffer-switch]]),
=hack-local-variables-hook= restores companion links
([[file:../lisp/ogent-companion.el::233][ogent-companion--restore-link]]), and =after-save-hook= can refresh
the codemap ([[file:../lisp/ogent-codemap.el::560][lisp/ogent-codemap.el:560]] — installed always, gated by
its defcustom inside). The edit pipeline decouples logging from display
through one hook: [[file:../lisp/ogent-edit-display.el::167][ogent-edit-resolved-hook]], consumed by
[[file:../lisp/ogent-edit-log.el::149][lisp/ogent-edit-log.el:149]].

** Generated functions and timers

MCP tools are =fset= at runtime
([[file:../lisp/ogent-mcp.el::325][ogent-mcp--register-tools]]) — =mcp-SERVER-NAME= functions exist only
after a server connects and are invisible to the byte-compiler. MCP
auto-connect runs on a one-second timer after ogent loads
([[file:../lisp/ogent-mcp.el::549][ogent-mcp-setup]]). The provider-fallback login offer is deferred via
=run-at-time 0= so it never prompts inside a gptel callback
([[file:../lisp/ogent-provider-fallback.el::101][ogent-provider-maybe-offer-login]]). Status spinners and the
header-line clock are repeating timers in
[[file:../lisp/ui/ogent-ui-status.el::135][lisp/ui/ogent-ui-status.el:135]].

** Where state actually lives

In-memory: the request table ([[file:../lisp/ui/ogent-ui.el::687][ogent-ui--request-table]]), the tool
registries ([[file:../lisp/ogent-models.el::202][ogent-tool-registry]] +
[[file:../lisp/ogent-models.el::264][ogent--tools-registered]] cache), pinned context
([[file:../lisp/ogent-context.el::87][ogent-pinned-context]]), MCP connections, tool-call history. On disk:
companion link registry (=user-emacs-directory=), sessions
(=~/.emacs.d/ogent-sessions/=), analytics (=.ogent-analytics.db= per
project), the ledger (=.ogent/ledger.org= per project), OAuth tokens
(mode-600 =tokens.el=), and the entire Armory (Org files under the
armory root; transient derivatives under =.armory-state/=). Nothing
