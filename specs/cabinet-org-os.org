#+title: Org Cabinet Integration

* Intent

ogent should keep its current identity as an Org-first LLM exoskeleton and gain
the useful Cabinet product layer: local knowledge base, AI team, durable
conversations, recurring jobs, task board, search, git history, and embedded
apps.

The port is not a TypeScript or Markdown migration. The durable format is Org.
The frontend is Emacs. The runtime surface is Elisp commands, Org buffers,
agenda/search buffers, magit-section style dashboards, and gptel-backed model
dispatch.

* Architectural Translation

| Cabinet concept | Cabinet implementation | ogent implementation |
|-----------------+------------------------+----------------------|
| Knowledge pages | Markdown files with frontmatter | Org files and subtrees with property drawers |
| Page editor | Tiptap in React | Org buffers, narrowing, sparse trees, capture, refiling |
| Sidebar tree | React file tree | Emacs tree/tabulated buffer over cabinet roots |
| Agents | =persona.md= plus jobs and memory dirs | =persona.org= plus Org job and memory files |
| Conversations | Markdown prompt/transcript plus JSON metadata | Org transcript files using headings, src blocks, tool blocks, and properties |
| Tasks board | React kanban over conversation/task state | Magit-section or tabulated-list board over Org task/conversation properties |
| Scheduler | node-cron daemon | Emacs timers for interactive mode; cron/automation integration for detached runs |
| Search | Flexsearch index | =consult-ripgrep= / =occur= / =org-ql= style adapters when present |
| Git history | Node simple-git routes | Magit/VC commands and optional automatic commits |
| Embedded apps | =index.html= iframe | =browse-url=, =xwidget-webkit=, or =eww= preview from an Org-linked app node |
| Provider runtime | CLI adapter registry | Codex and Claude CLI runner adapters plus gptel-backed interactive dispatch |

* Org Storage Contract

All new Cabinet-style records use Org:

#+begin_example
cabinet-root/
  index.org
  .agents/
    .conversations/
    editor/
      persona.org
      jobs/
        weekly-review.org
      memory/
        context.org
        decisions.org
        learnings.org
      sessions/
  .jobs/
  .cabinet-state/
#+end_example

** Cabinet Index

=index.org= marks a cabinet root:

#+begin_src org
#+title: Acme Research

* Acme Research
:PROPERTIES:
:OGENT_CABINET: t
:OGENT_CABINET_ID: acme-research-root
:OGENT_KIND: root
:OGENT_DESCRIPTION: Research lab
:OGENT_TAGS: ai, ops
:END:
#+end_src

** Agent Persona

=.agents/<slug>/persona.org= is the source of truth for identity and durable
instructions:

#+begin_src org
#+title: CTO

* CTO
:PROPERTIES:
:OGENT_AGENT: t
:OGENT_SLUG: cto
:OGENT_ROLE: Architecture
:OGENT_PROVIDER: codex
:OGENT_MODEL: gpt-5.4
:OGENT_PERMISSION_MODE:
:OGENT_HEARTBEAT: 0 9 * * 1-5
:OGENT_ACTIVE: t
:OGENT_WORKSPACE: engineering
:OGENT_TAGS: engineering, strategy
:END:

Keep the technical plan brutally clear.
#+end_src

** Agent Job

=.agents/<slug>/jobs/<job-id>.org= stores a recurring prompt as an Org task:

#+begin_src org
#+title: Weekly Review

* TODO Weekly Review
:PROPERTIES:
:OGENT_JOB: t
:OGENT_JOB_ID: weekly-review
:OGENT_AGENT: editor
:OGENT_CRON: 0 9 * * 1
:OGENT_HEARTBEAT:
:OGENT_PROVIDER:
:OGENT_MODEL:
:OGENT_WORKSPACE:
:OGENT_TAGS: review, weekly
:OGENT_ENABLED: t
:OGENT_ARCHIVED: nil
:END:

Review recent Org pages and file a summary.
#+end_src

** Running Agents

=ogent-cabinet-runner.el= turns persona and job records into concrete CLI
process plans:

- =OGENT_PROVIDER= selects the runner. Supported values normalize to =codex=
  or =claude=.
- Codex agents run through =codex exec= with the prompt on stdin and inherit the
  local =codex login= session. Cabinet runs pass =--skip-git-repo-check= by
  default because a knowledge root may not be a git checkout.
- Claude agents run through =claude -p= and inherit the local Claude Code
  first-party login.
- =OGENT_MODEL= passes a model override to the selected CLI when present.
- =OGENT_PERMISSION_MODE= passes the Claude Code permission mode when present.
- =OGENT_WORKSPACE= chooses the process working directory relative to the
  cabinet root, or an absolute directory.

Every manual run and job run writes an Org transcript under
=.agents/<slug>/sessions/=. The transcript stores provider, model, job,
workspace, duration, app paths, exit status, prompt, stdout, and stderr in
property drawers and src blocks.

* Graph and Projection Model

Cabinet storage is not just a file layout. It is the base relation layer for an
Org-native operational graph:

#+begin_example
Cabinet(id, path, name, kind)
Agent(slug, path, name, role, provider, active)
Job(agent_slug, job_id, path, name, cron, heartbeat, enabled)
Session(agent_slug, session_id, path, status, provider, model)
App(path, owner)
Issue(id, worker)
GasTownHook(root)

contains        : Cabinet -> Agent
owns            : Agent   -> Job
owns            : Agent   -> Session
scheduled-by    : Job     -> Session
produced        : Session -> App
failed-from     : Job     -> failed Session
archived-from   : Agent/Job/Session -> Cabinet
linked-issue    : Record  -> Issue
assigned-worker : Issue   -> Agent/GasTown worker
#+end_example

=ogent-cabinet-build-graph= materializes the derived graph from Org files.
Nodes are plists with =:id=, =:kind=, =:label=, =:path=, and =:data=. Edges are
plists with =:from=, =:to=, and =:kind=. The graph is rebuildable from
=index.org=, persona files, job files, transcripts, issue properties, discovered
apps, and local Gas Town markers.

This shape makes the Cabinet layer a natural extension of existing ogent
systems:

- =ogent-issues= can project beads work into the same graph by attaching issue
  nodes to cabinet, agent, job, or conversation nodes.
- =ogent-gastown-status= can project hooks, mail, convoys, rigs, and worker
  assignments as operational edges.
- Org agenda/search already performs collection and display over scattered TODO
  entries, tags, and properties, which matches the projection model.
- Future indexes can move from full rebuilds toward change propagation over Org
  edits, following the same broad family of ideas as differential dataflow and
  incremental lambda calculus.

The primary projection surface is =ogent-cabinet-home=. It summarizes Cabinet
metadata, health, navigation, recent activity, and attention items. The graph
surface remains available through =ogent-cabinet-status= and is keyboard
navigable with =RET= for source visits, =R= for runnable nodes, =e=/=E= for
metadata and body edits, =P=/=J=/=C= for agent profile, jobs, and job creation,
=m= for its Transient menu, and jumps to the richer surfaces with =a=, =t=,
=c=, =s=, =A=, and =G=.

* UI Contract

The Emacs surfaces are boring and powerful:

- =ogent-cabinet-scaffold= creates an Org-backed cabinet directory.
- =ogent-cabinet-home= opens the operational home view with Cabinet metadata,
  health counts, navigation, recent activity, app artifacts, and needs-attention
  sections for failed sessions, stale jobs, missing persona fields, and open
  conversations. Header-line keys are visible: =RET=, =g=, =q=, =a=, =t=,
  =c=, =s=, =A=, =G=, =e=, =n=, and =p=.
- =ogent-cabinet-status= shows the derived Cabinet graph and operational
  bridges to Issues and Gas Town. Press =R= on an agent, job, or retryable
  session to run it through the selected CLI provider. Press =e= to edit
  metadata in the durable Org property drawer, =E= to edit a body or job
  prompt, =P= for an agent profile, =J= for the selected agent's jobs, =C= to
  create a job for the selected agent, =m= for the Transient menu, and =?= for
  help.
- =ogent-cabinet-agents= shows personas in a tabulated buffer with sortable
  columns for name, slug, role, provider, model, active state, job count,
  conversation count, last run, workspace, and tags.
- =ogent-cabinet-agent= opens a single agent profile buffer with sections
  for composer, inbox, conversations, recent work, schedule, memory,
  tools/permissions, details, and persona instructions. The profile header makes
  =RET=, =c=, =e=, =R=, =v=, and =q= visible.
- =ogent-cabinet-create-agent=, =ogent-cabinet-clone-agent=, and
  =ogent-cabinet-archive-agent= manage agent lifecycle through Org files.
- =ogent-cabinet-jobs= and =ogent-cabinet-agent-jobs= list jobs globally or for
  one agent. =ogent-cabinet-create-job= writes job records under the owning
  agent's =jobs/= directory. Job actions cover edit, enable/disable, run,
  retry, archive, and source visits.
- =ogent-cabinet-tasks= shows attention-first lanes: Inbox, Needs Reply,
  Running, Scheduled, Just Finished, Stale, and Archive.
- =ogent-cabinet-conversations= lists all transcripts and filters by agent, job,
  status, provider, model, tag, and archive state. =RET= opens a detail buffer
  with metadata, prompt, output, tool blocks, exit status, duration, linked
  agent/job, and source Org.
- =ogent-cabinet-search= searches Cabinet Org files, personas, jobs, memory,
  inboxes, schedules, imports, and conversations. Empty queries display a useful
  empty state. Search skips =.git= and transient state directories.
- =ogent-cabinet-apps= discovers =index.html= artifacts, shows owning
  agent/job/session when inferable, opens artifacts in a browser, and visits app
  directories.
- =ogent-cabinet-import-artifacts= imports Markdown, HTML, and text artifacts as
  Org records under the Cabinet while preserving originals by default.

The implementation should use existing ogent primitives whenever possible:

- Context building stays in =ogent-context.el=.
- Model selection stays in =ogent-models.el= and gptel integration.
- Conversation rendering should reuse current Org src/tool block conventions.
- Tool effects and approval policy stay in =ogent-tool-effects.el= and
  =ogent-tool-approval.el=.
- Beads and Gas Town stay as their own operational backends while Cabinet
  records expose typed graph attachment points.

* Phases

** Phase 1: Org Storage Foundation

- Add =ogent-cabinet.el=.
- Scaffold cabinet directories with =index.org=.
- Write/read =persona.org= agent files.
- Write/read Org job files.
- Build a typed graph projection over cabinet, agent, job, session, app, issue,
  and Gas Town hook records.
- Add Cabinet home and status buffers that bridge to Issues and Gas Town.
- Test that no Markdown files are emitted.

** Phase 2: Agent Profile Buffers

- Add a read-only profile buffer for one agent.
- Add commands to edit identity fields via Org property drawers.
- Add a composer entry point that dispatches through current ogent request
  machinery.
- Run agents and jobs through Codex or Claude CLI providers from the status
  buffer.

** Phase 3: Conversation Store

- Store every manual run, job run, and heartbeat run as one Org conversation
  file with metadata properties.
- Preserve current response src block and tool block rendering.
- Add resume metadata without provider-specific sidecars.
- Parse transcript detail views from Org files.
- Link generated app artifacts back to sessions when runs produce apps.

** Phase 4: Schedule and Task Board

- Add agenda-compatible views for enabled jobs and recent conversations.
- Add an attention board with Cabinet lanes over =tabulated-list-mode=.
- Add retry/resume/archive commands over Org metadata.
- Add malformed metadata validation with friendly errors where job metadata is
  edited or run.

** Phase 5: Search, Git, Apps

- Add cabinet-scoped search commands.
- Add optional Magit/VC history commands for changed Org files.
- Add app artifact preview commands for directories with =index.html=.
- Add a practical importer for Markdown, HTML, and text artifact folders.
- Document migration limits and the first-ten-minutes workflow.

* Open Decisions

- Whether cron execution should live only in Emacs timers or also expose a
  detached automation bridge.
- The Cabinet task board starts on =tabulated-list-mode= for dense keyboard
  sorting; a later Magit-section projection can be added if nested sections
  become important.
- Whether the graph projection should gain an on-disk incremental cache, and
  which Org edit hooks should feed its change set.
- Whether org-roam should become the recommended cross-cabinet index or remain
  optional.
