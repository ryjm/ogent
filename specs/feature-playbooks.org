#+title: ogent Feature Playbooks

* Using This File
Always review the relevant playbook before starting a feature. Each section links back to =specs/architecture.org= for shared constraints.

* Prompt Dispatcher Enhancements
- Confirm whether the change belongs in =lisp/ui/ogent-ui.el= (transient layout) or the core context pipeline.
- Update =ogent-prompt-dispatch= buttons via dedicated functions (never inline lambdas) so they remain autoloadable.
- Tests: extend =test/ui/ogent-ui-tests.el= with scenario coverage. Use =ogent-test-with-fixture= and assert buffer mutations.

* Speed Coding Edits
- Focused code edit commands live in =lisp/ogent-edit.el= and reuse the SEARCH/REPLACE pipeline.
- AI speed edit commands should summarize editor state, target scope, point location, and diagnostics before asking the model to choose a small patch.
- Diagnostic-driven commands should normalize editor sources such as Flymake and Flycheck before prompt construction.
- Buffer-wide repair commands should rank and deduplicate diagnostics before asking for a full-buffer edit.
- Bind new user-facing commands through =lisp/ogent-keys.el= for vanilla, Evil, and Doom parity.
- Document the final key chord in =README.md= and =docs/getting-started.md=.
- Tests: extend =test/ogent-edit-tests.el= for target selection and =test/ogent-keys-tests.el= for registry coverage.

* Context Resolution Changes
- Work inside =lisp/ogent-context.el= and add mirrored tests in =test/ogent-context-tests.el=.
- Maintain plist contracts: =:root=, =:ancestors=, =:handles=, =:dependencies=.
- Handle new metadata via property drawers; never parse raw text beyond =org-element= APIs.

* Codemap Updates
- Changes live in =lisp/ogent-codemap.el=; keep output strictly Org headings with file links.
- Tests must assert both definition discovery and buffer mode.

* Documentation & Specs
- Store human-facing guides in =docs/=; architectural decisions stay here under =specs/=.
- When adding a new feature, drop a short playbook entry describing where code, tests, and docs belong.

* Org Armory Features
- Storage helpers live in =lisp/ogent-armory.el=.
- CLI execution helpers live in =lisp/ogent-armory-runner.el=. Keep provider
  adapters small, explicit, and covered by fixture-backed process tests.
- The graph status surface lives in =lisp/ogent-armory-status.el= and should
  reuse =ogent-ops-style= plus navigation conventions from Issues.
- Agent profile and task-board UI should live under =lisp/ui/= once they need
  richer interaction than the graph status buffer.
- Durable records must be Org files: =index.org=, =persona.org=, job =.org=
  files, conversation transcript =.org= files.
- Do not introduce TypeScript, Markdown frontmatter, or a database layer for
  Armory records.
- Tests belong in =test/ogent-armory-tests.el= for storage contracts,
  =test/ogent-armory-runner-tests.el= for CLI runner plans and transcripts,
  =test/ogent-armory-status-tests.el= for graph status behavior, and =test/ui/=
  for richer interactive buffers.

* Structured Edit Outputs
- The gptel =:schema= path lives in =lisp/ogent-edit-parse.el= (pure schema + conversion) and =lisp/ogent-edit.el= (request wiring, =ogent-edit-use-structured-output=, backend capability probe).
- New payload fields extend =ogent-edit-structured-schema= AND the conversion into edit structs together; keep parity with the text parser (parity tests in =test/ogent-edit-parse-tests.el=).
- Fallback to text parsing must stay automatic; never make structured output load-bearing.

* Capture Templates
- Templates live in =ogent-capture-templates= (=lisp/ogent-notes.el=); registration only via =ogent-notes-setup-capture= (deferred, idempotent, never clobbers user keys).
- New targets get a file defcustom defaulting under =org-directory=.
- Tests in =test/ogent-notes-tests.el=: exactly-once registration, collision respect, target resolution. User docs: =docs/getting-started.md= Optional Features.

* Conversation Export (ox-ogent)
- Backends and filters live in =lisp/ox-ogent.el=; shared parse-tree filter strips =OGENT_*= drawers and relabels exchange headlines - extend the filter, not per-backend hacks.
- Interactive entry points: =ogent-export-conversation= and the =C-c C-e g= dispatcher wrappers; all must climb to the conversation root.
- Tests in =test/ox-ogent-tests.el= (string fixtures, no file writes). User docs: =docs/getting-started.md=.

* Armory Search Views (org-ql)
- The adapter lives in =lisp/ogent-armory-ql.el=: pure intent->sexp builder mirroring =ogent-armory-record-metadata= kinds, plus soft-required org-ql commands.
- New record kinds/statuses extend BOTH the metadata classification and =ogent-armory-ql-kind-properties=; saved views go in =ogent-armory-ql-saved-views=.
- Query-builder tests need no org-ql; command tests assert the install-hint error when it is absent. User docs: =docs/armory.org=.

* Native gptel Agent Runs
- The in-process loop lives in =lisp/ogent-armory-native.el=; the runner dispatches on adapter type =gptel_native= and everything funnels through the shared plan/finalize conversation lifecycle.
- Tool execution MUST go through =ogent-ui--execute-tool= (ledger) with =ogent-tool-approval-check= in front, and successful rounds through =ogent-analytics-record-completion= - never bypass the audit path.
- Tests stub =gptel-request= via =cl-letf= (=test/ogent-armory-native-tests.el=). User docs: =docs/armory.org=.
