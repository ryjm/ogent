#+title: ogent Architectural Constraints

* Design Philosophy

ogent treats language models as Unix tools: they take structured input, produce structured output, and compose with everything else in Emacs. The user remains in the loop, wielding the tool rather than supervising an agent.

** Core Principles

*** Text as Interface
Everything flows through Org. Context is text. Responses are text. Diffs are text. This means standard Emacs operations—search, replace, version control, export—work on LLM interactions the same as any other buffer content.

*** Explicit Context
No implicit "send my whole project." The user declares what goes into the context window:
- =@handle= syntax resolves to specific Org nodes, files, or org-roam entries
- =ogent-codemap= generates a navigable index of the codebase
- =ogent-pin-dwim= persists items across requests
- =ogent-context-preview= shows exactly what will be sent

This explicitness serves two purposes: reproducibility (same input → same behavior) and efficiency (smaller context → faster responses, lower cost).

*** Responses as Data
Model output lands in Org src blocks with structured headers:
#+begin_example
#+begin_src text :model claude-sonnet-4-6 :status done :latency 2.1s
#+end_example

This metadata enables programmatic access. Fold responses, navigate between them, extract them for export. The response is not ephemeral chat—it's a first-class Org element.

*** Diffs Over Descriptions
When the model suggests code changes, ogent renders them as smerge conflicts:
- Red/green highlighting shows exactly what changed
- =C-c C-c= accepts, =C-c C-k= rejects
- No parsing prose like "change line 42 to..."

Visual review is faster than textual review. The interface should optimize for the bottleneck, which is human verification.

** Technical Implementation

*** Context Pipeline
1. =ogent-context-build= resolves the current subtree, ancestors, and =@handle= dependencies
2. =ogent-context-build-lazy= defers expensive resolution until needed
3. Context flows as an immutable plist—presentation code never mutates it
4. =ogent-response-function= fires exactly once per request for reliable hook execution

*** Model Orchestration
- =ogent-model-registry= declares backends with =:id=, =:backend=, =:stream?=, and optional =:preset=
- =ogent-prompt-dispatch= can fan out to multiple models simultaneously
- =ogent-ui--request-table= tracks concurrent streams for cancellation/retry
- gptel handles transport; ogent handles orchestration and presentation

*** Companion Buffers
When invoked from non-Org buffers, ogent creates a companion Org buffer for the transcript:
- Code buffer receives inline diffs/patches
- Companion buffer records prompts, context, and responses
- Markers keep the two synchronized
- Navigation commands work from either buffer

*** Tool Execution
- =ogent-tool-registry= defines available tools with approval workflows
- Tool results render in =#+begin_tool= blocks, folded by default
- =ogent-tool-fsm= manages the approval state machine
- Users can re-run tools at point with =ogent-tool-rerun=

*** Proof Ledger
Requests and tool calls may be recorded through =ogent-ledger.el= when
=ogent-ledger-enabled= is non-nil. Ledger entries are append-only Org
headings with event type, hash metadata, and printable event data.

The ledger is intentionally explicit:
- Request entries store prompt and context hashes, not full prompt payloads.
- Tool entries store declared effects, result hashes, errors, and duration.
- Buffer and marker values are sanitized to printable references.

This gives ogent a replayable audit trail without hidden memory.

*** Org Armory Layer
Armory-style knowledge-base and AI-team features live behind an Org storage
contract. The durable records are Org files, not JSON databases or Markdown
frontmatter:
- armory roots use =index.org= with =OGENT_ARMORY= properties
- agents use =.agents/<slug>/persona.org=
- recurring jobs use =.agents/<slug>/jobs/<job-id>.org=
- conversations should use Org transcript files with the same src/tool block
  conventions as normal ogent requests
- =ogent-armory-build-graph= materializes those files into a typed directed
  graph with =armory=, =agent=, and =job= nodes plus semantic edges such as
  =contains= and =owns=
- =ogent-armory-status= renders that graph as the Armory operational entry
  point and links into =ogent-issues= and =ogent-gastown-status=

See [[file:armory-org-os.org][Org Armory Integration]] for the full mapping from
Armory's repository architecture to ogent's Emacs/Org architecture.

*** Tool Effects
Every tool spec should declare =:effects=. Effects are plists with:
- =:kind= such as =read=, =write=, =execute=, =network=, =git=,
  =issue=, or =emacs-state=
- =:target= such as =file=, =filesystem=, =shell=, or =external=
- =:scope= such as =workspace=, =process=, =command=, or =unrestricted=
- =:risk= such as =low=, =medium=, =high=, or =critical=

Approval policy must consider effects, not only tool names. Read-only
workspace tools can run without prompting by default, while write,
execute, network, git, issue, and Emacs-state effects require approval.

* Rule of Engagement
- Treat this file as required context before building any feature.
- Emacs Lisp modules must stay under =lisp/=, UI helpers under =lisp/ui/=, docs under =docs/=, specs here.
- Every new command exported to users needs an =ogent-= prefix and an associated test in =test/= mirroring the namespace.

* Minor Mode Contract
- =ogent-mode= owns the =C-c o= prefix; do not bind global keys directly.
- Autoload interactive entrypoints so Doom packaging can lazily load.
- Keep state local to buffers: use buffer-local variables or plists on Org properties.

* Context Pipeline
1. Resolve the current subtree via =ogent-context-build=.
2. Collect ancestor headings, handles, and dependency plists.
3. Pass the plist unchanged into the UI layer; presentation code must not mutate nodes.
4. All model dispatchers must call =ogent-response-function= exactly once per request so hooks fire reliably.

* Codemap & Discovery
- Codemap output => Org subtree under a dedicated buffer to keep navigation consistent.
- When adding new namespaces, update =ogent-codemap-source-directories= and write tests proving definitions are discovered.

* Testing Expectations
- Use =ogent-test-with-fixture= when Org buffers are involved.
- For UI commands, assert both structural changes (e.g., src block insertion) and guard-rail behavior (model toggles, preview buffer creation).
- Always run =make test= and =make lint= before committing.

* Model Registry & Transport
- Declare every provider in =lisp/ogent-models.el=. Each entry must include =:id=, =:backend=, and =:stream?=. Optional keys (=:preset=, =:description=) feed the UI and docs.
- `ogent-prompt-dispatch` is generated from the registry. Never hard-code model buttons; call =ogent-ui-refresh-dispatch= after changing the registry so tests see the new toggle set.
- All transport code is expected to go through gptel (see [[file:gptel/overview.org][gptel overview]] and [[file:gptel-integration.org][integration plan]]). No direct HTTP clients inside ogent.
- Keep =ogent-model-registry= and =ogent-gptel-required-features= in sync so `ogent-ui--ensure-gptel` can `require` the right backend feature before streaming.

* Streaming & Error Handling
- `ogent-response-function` must return an =ogent-ui-request= struct. The UI layer is responsible for inserting placeholder Org blocks before gptel fires.
- Streaming callbacks append to the existing block and close it by writing =#+end_src= when gptel signals done/abort. Errors must produce a =#+begin_quote ogent-error= block without deleting prior context.
- Maintain the request table (=ogent-ui--request-table=) so cancellation/resume features can target active streams.
- Hooks (`ogent-after-request-hook`) are run exactly once per request—after blocks close successfully or with error annotations.

* Buffer Linking Outside Org-mode
- Invoking ogent from a non-Org major mode must establish (or reuse) a companion Org buffer that holds the request/response transcript. By default, create a temp Org buffer named after the source buffer (e.g., `*ogent:<file>*`) and store the link in buffer-local metadata so follow-up commands reuse the same companion.
- Users can override the companion by selecting an existing Org buffer (prompt or config). Once linked, ogent commands route context reads to the originating buffer and writes/streamed responses to the Org companion.
- All editing actions must respect the active context: code buffers receive inline diffs/patches when the user asks for edits, while the Org companion records prompts, context summaries, and streamed completions. Keep the two buffers synchronized via markers so navigation commands (e.g., jump to last response) work regardless of where ogent was invoked.
- Ensure the linking metadata survives buffer reloads (e.g., via buffer-local vars saved in file-local settings) and expose a command to rebind the companion Org buffer when users switch workflows.

* Proof Ledger & Effects
- =ogent-ledger.el= owns append-only request and tool event recording.
- =ogent-tool-effects.el= owns effect normalization, formatting, risk
  scoring, and approval policy primitives.
- Ledger recording is gated by =ogent-ledger-enabled=.
- New tools must declare =:effects= in their registry spec.
- Approval UI should display declared effects before arguments.
- Context manifests should include hashes and reasons for every included
  source, Org node, handle, and pinned item.

If a feature introduces new architectural rules, add a child heading here or create another file under =specs/= and cross-link it.
