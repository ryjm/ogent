#+title: Cabinet Parity Specification for Org and Emacs
#+date: 2026-05-06

* Purpose

This document specifies the Cabinet functionality ogent already has, the
Cabinet functionality observed in the official Cabinet app =~/vault/projects/runcabinet=,
and the work required to reach product parity in an Org-mode and Emacs context.

The target is not a React clone. Cabinet's useful product ideas should become
native Emacs workflows backed by Org files. Org remains the durable record,
Emacs remains the operating surface, and local CLI agents remain first-class
runtimes.

The running app at =localhost:4000= was exercised with desktop automation. The

  work, schedule, details, skills, and persona editor.
- Tasks board with Kanban/List/Schedule modes, density, cabinet scope, agent
  filter, trigger chips, task composer, lanes, card actions, and detail panel.

Observed issue in the live app: opening one =Researcher= inbox task displayed a
right-side detail panel, then stayed in a loading state during this session.
The spec treats the intended task-detail behavior as source-truth from the
Cabinet code and PRDs, with this live hang recorded as a QA note.

* Current Ogent Cabinet State

** Durable Storage Already Exists

Ogent already treats Cabinet data as Org records:

#+begin_example
cabinet-root/
  index.org
  imports/
  .agents/
    <slug>/
      persona.org
      inbox.org
      schedule.org
      jobs/
        <job-id>.org
      memory/
        context.org
        decisions.org
        learnings.org
      sessions/
        <timestamp>-<job-or-run>.org
  .jobs/
  .cabinet-state/
#+end_example

Implemented storage capabilities in =lisp/ogent-cabinet.el=:

- Scaffold a Cabinet root with =index.org= and managed directories.
- Read and write Cabinet index metadata.
- Read and write agent personas as =persona.org=.
- Create per-agent =jobs=, =memory=, =inbox.org=, =schedule.org=, and
  =sessions= directories.
- Read and write jobs as Org files.
- Validate job metadata, including cron, heartbeat, enabled, archived, and
  provider fields.
- Read session transcript Org files.
- Parse prompt, output, error, runtime trace, and tool blocks from session
  transcripts.
- List sessions across agents.
- Search Org records with filters for kind, agent, status, tags, archive state,
  and record body.
- Discover generated app artifacts by scanning for =index.html=.
- Build a typed operational graph containing cabinet, agent, job, session, app,
  issue, and Gas Town hook nodes.
- Import =.md=, =.html=, and =.txt= artifacts into Org import records.

** Runner Already Exists

Implemented runner capabilities in =lisp/ogent-cabinet-runner.el=:

- Provide auth status helpers for Codex and Claude.

** Current Gaps Inside Ogent

The main parity gaps are architectural, not cosmetic:

- Conversations are saved as per-agent session transcripts under
  =.agents/<slug>/sessions/=. Cabinet v2 uses a canonical global
  =.agents/.conversations/<id>/= store with metadata, turns, events, artifacts,
  attachments, and session handles.
- Ogent has single-run transcripts, but not durable multi-turn continuation.
- Ogent has no conversation event log, reconnect replay, or live task update
  bus.
- Ogent supports Codex and Claude CLI only. Cabinet has eight provider families
  and a structured adapter interface.
- Ogent has no runtime picker abstraction for provider, adapter, model, effort,
  and terminal/native mode.
- Ogent has no provider verification, install guidance, dynamic model listing,
  or error taxonomy.
- Ogent has no skill catalog, skill mention, per-agent skill selection, or
  runtime skill mount.
- Ogent has no attachment staging or Org attachment contract for prompts.
- Ogent has no Cabinet-scoped nested cabinet visibility model.
- Ogent has no onboarding wizard, registry import, or team template flow.
- Ogent has no native command palette across Cabinet records.
- Ogent has no schedule calendar view.
- Ogent has no git dirty popover, commit, pull, log, diff, or restore UI.
- Ogent has no terminal tab manager or PTY bridge.
- Ogent has no settings surface for providers, storage, integrations,
  notifications, appearance, updates, or data locations.
- Ogent has no page file manager with move, reorder, cabinet creation, or
  hidden-file toggles.
- Ogent has no WYSIWYG editor parity, media viewers, PDF/CSV/Office viewers, or
  rich page embeds - but we do not need that functionality.

* Cabinet Product Inventory

** Application Shell and Navigation

Cabinet's shell combines:

- Persistent sidebar with drawers for =Data=, =Agents=, and =Tasks=.
- Collapsible and resizable drawer.
- Cabinet scope and visibility controls.
- Data tree with nested pages, folders, and cabinets.
- Drawer footers with =New Page=, =New Cabinet=, =New Task=, settings, refresh,
  and collapse actions.
- Main content router for page editor, cabinet view, agents workspace, agent
  detail, tasks board, task conversation, settings, registry, help, and
  onboarding.
- Status bar with save state, AI edit prompt, runtime picker, git status,
  pull, terminal, help/community, and app/daemon health.
- Global command palette on =Mod-k=.

Emacs target:

- Keep =ogent-cabinet-home= as the main landing surface.
- Add =ogent-cabinet-sidebar= as an optional persistent side window or
  consult-driven navigation source.
- Add =ogent-cabinet-command= as a single Transient/Completing-read command
  palette across pages, agents, tasks, jobs, conversations, apps, settings, and
  git operations.
- Keep drawer parity through commands and buffers, not a forced three-pane UI.

** Data and Page Editing

Cabinet pages are Markdown files with Tiptap editing, Markdown source toggle,
frontmatter metadata, wiki links, internal links, uploads, slash commands,
tables, checklists, callouts, math, emoji, media, embeds, heading anchors,
drag handles, resizable images, color, highlight, alignment, and RTL metadata.

Cabinet also provides viewers for source files, images, media, websites,
notebooks, CSV, PDF, Google Docs, Mermaid, DOCX, XLSX, PPTX, and folders.

Emacs target:

- Org editing is already richer than Cabinet for hierarchy, TODOs, narrowing,
  refiling, capture, agenda, links, source blocks, tables, exports, and local
  scripting.
- Add a Cabinet data browser over Org files and imported assets.
- Add =ogent-cabinet-page-create=, =ogent-cabinet-page-rename=,
  =ogent-cabinet-page-move=, =ogent-cabinet-page-delete=, and
  =ogent-cabinet-page-export=.
- Add per-page metadata commands for =OGENT_TITLE=, =OGENT_KIND=,
  =OGENT_CABINET=, =OGENT_DIR=, =OGENT_TAGS=, and optional display fields.
- Use Org links for wiki links and internal links. Investigate whether org-roam
  would be a better fit as well.
- Use =org-attach= for page attachments.
- Use existing Emacs major modes for source, CSV, image, PDF, and media where
  available.
- Use =browse-url=, =eww=, or =xwidget-webkit= for websites and generated apps.
- Treat rich WYSIWYG toolbar parity as non-goal for the first parity release.
  Org's native editing commands are the primary page editor.

Important Cabinet conversation fields:

  times, exit code, job id/name, scheduledAt, provider, adapter, config, prompt
  path, transcript path, mentions, attachments, artifacts, summary, context

  archived, awaiting input, pinned title, board order, muted, error kind, error
  hint, retry-after, last resume attempt, parent task id, triggering agent,

Cabinet task board maps conversations into lanes:

Cabinet task detail includes:

- Status action, stop, restart, close/archive, copy link, delete.
- Turn list with user and agent blocks.
- Continue composer with attachments, mentions, runtime picker, and scheduling.
- Terminal view for terminal-mode runs.
- Claude transcript view for compatible sessions.
- Artifacts, diffs, logs, approval panel, token bar, compact, wrap-up card,
  pending actions, and schedule/restart dialogs.

Ogent has:

- Per-agent session transcripts.
- Conversation list and detail over those session files.
- Retry and archive/unarchive metadata.
- Task board derived from jobs, sessions, and live subprocesses.

** Runtime Providers and Adapters

* What Emacs Can Do Better

** Org-Native Knowledge Work

Org gives Cabinet features a stronger base:

- Headings are durable objects with properties, TODO states, tags, links, and
  timestamps.
- Agenda collects scheduled jobs, TODOs, and timestamped work across files.
- Properties act as a lightweight database inside normal text files.
- Org links cover files, headings, URLs, emails, and custom protocols.
- Org attachments give each conversation, page, or task a filesystem-backed
  attachment directory.
- Org export can cover HTML, Markdown, LaTeX/PDF, and publish flows.

** Emacs Interaction Model

Emacs improves repeated Cabinet workflows:

- Minibuffer completion scales better than a browser dropdown for thousands of
  pages and tasks.
- Magit already solves deep git workflows.
- Dired already solves file movement, marking, copying, and bulk operations.
- Transient gives keyboard-native menus for complex commands.
- tabulated-list and magit-section give dense, sortable, collapsible buffers.
- Embark actions can make every search result actionable.
- TRAMP can let a Cabinet root live on a remote machine.

** Agent Workflows

Emacs is a better host for text-first agent sessions:

- Conversation transcripts can be edited, narrowed, refilled, searched, and
  linked directly.
- Source code, Org plans, terminal buffers, diffs, and agent transcripts can
  sit in one window configuration.
- gptel can remain available from any buffer for lightweight LLM work.
- CLI agents can remain separate for file-editing and resume-heavy tasks.

** Boundaries Where Browser Cabinet Stays Stronger

Some Cabinet surfaces are naturally richer in the browser:

- WYSIWYG drag handles and visual embeds.
- Inline media layout controls.
- Full iframe app previews.
- PDF/Office viewers with polished chrome.
- Pointer-first Kanban dragging.
- Animated onboarding and visual demos.

Emacs parity should prioritize durable behavior and operational power. Visual
parity is useful only where it improves the workflow.

** Storage Rules

- Org files are the durable source.
- JSON may be exported for compatibility, but it must be derived.
- All records need stable ids.
- All mutable state must be editable by hand without corrupting the reader.
- Every reader must tolerate older records.
- Every writer must preserve unknown properties.
- All date fields use ISO 8601 strings.
- All file paths stored in properties are cabinet-relative unless explicitly
  marked as absolute.

** Conversation Lifecycle

1. User runs =ogent-cabinet-compose= from home, agent, task, page, or command
   palette.
2. Composer resolves:
   - cabinet scope
   - agent
   - prompt
   - mentioned pages
   - mentioned agents
   - mentioned skills
   - attachments
   - runtime override
   - schedule mode
3. For run-now:
   - Create =.agents/.conversations/<id>/index.org=.
   - Move staged attachments.
   - Append first user turn.
   - Build adapter prompt.
   - Start native or terminal process.
   - Mark status =running=.
   - Append events.
4. On process output:
   - Append transcript chunks.
   - Parse cabinet metadata block.
   - Parse ask-user marker.
   - Parse agent actions.
   - Update artifacts, summary, context, tokens, and status.
   - Append agent turn.
   - Mark =awaiting-input=, =done=, or =failed=.
5. For continuation:
   - Append user turn.
   - Resume when possible.
   - Replay prior turns when resume is unavailable.
   - Record resume path.
6. For archive/delete:
   - Archive toggles properties.
   - Delete moves to =.cabinet-state/trash/= by default, with hard delete as an
     explicit command.

- Whether global agents live inside each Cabinet tree, a user-global directory,

- Whether schedule execution should remain Emacs-timer based for parity v1 or

  built-in =comint=.
