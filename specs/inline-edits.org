#+title: ogent Inline Edits Specification
#+author: Jake Miller
#+date: 2025-12-09

Revised 2026-07-13 to match the shipped implementation: three display
methods, diff preview/review buffers, the quick-edit / diagnostic-repair /
AI-speed-edit request flows, and the registry-driven keybindings.

* Purpose
This spec defines how ogent applies LLM-generated code changes to source buffers.
When users request edits from code files (not Org buffers), ogent should:
1. Send the code context to the LLM with edit-format instructions
2. Parse the structured edit response
3. Display proposed changes inline using the configured display method
   (smerge conflict markers, overlays, or inline word-level diffs)
4. Allow accept/reject workflow, per edit or in batch
5. Log all operations to the companion Org buffer

* Design Principles
- *Leverage built-ins*: =smerge-mode= (ships with Emacs) is the default and
  always-available display; richer methods layer on top
- *Configurable review*: display method is a user choice
  (=ogent-edit-display-method=), togglable mid-session
- *Safe by default*: no auto-apply unless =ogent-edit-auto-apply= is
  explicitly enabled; user must accept changes
- *Robust against drift*: apply paths re-anchor an edit's cached positions
  before touching the buffer, so edits survive buffer changes between
  validation and application
- *Auditable*: every edit proposal logged to companion with timestamps
- *Reversible*: rejected edits preserved in history; auto-applied edits
  undoable with standard Emacs undo
- *Streaming-aware*: handle partial responses during streaming

* Edit Request Format

** Search/Replace Blocks (Primary)
The LLM returns edits as SEARCH/REPLACE blocks. This format is:
- Unambiguous: exact text matching
- Multi-hunk: supports multiple changes per response
- Validated: OLD text must exist exactly in source

#+begin_example
To make the requested changes, I'll modify the following:

<<<<<<< SEARCH
(defun foo ()
  "Old docstring."
  (old-implementation))
=======
(defun foo ()
  "New improved docstring."
  (new-implementation))
>>>>>>> REPLACE

<<<<<<< SEARCH
(setq bar 42)
=======
(setq bar 100)
>>>>>>> REPLACE
#+end_example

** Format Rules
1. Each edit block starts with =<<<<<<< SEARCH=
2. Original code follows (must match source exactly)
3. Separator is ========
4. New code follows
5. Block ends with =>>>>>>> REPLACE=
6. Multiple blocks allowed per response
7. Surrounding prose/explanation is allowed and ignored by parser

** Why This Format?
| Format         | Pros                          | Cons                                  |
|----------------+-------------------------------+---------------------------------------|
| Unified diff   | Standard, line-based          | Fragile with LLM, line numbers drift  |
| JSON           | Structured, machine-friendly  | Verbose, hard for LLM to format       |
| SEARCH/REPLACE | Simple, exact matching, Claude-native | Requires unique matches       |

SEARCH/REPLACE is chosen because:
- Claude and other LLMs naturally produce this format
- Exact text matching avoids line number issues
- Simple regex parsing
- Used successfully by Aider, Claude Code, gptel-aibo

* Prompt Template

** System Instructions
=ogent-edit-system-prompt= (in =lisp/ogent-edit-format.el=) is passed as the
system prompt for every edit request:

#+begin_example
When making code changes, format each edit as a SEARCH/REPLACE block:

<<<<<<< SEARCH
[exact original code to find]
=======
[replacement code]
>>>>>>> REPLACE

Rules:
- The SEARCH section must match the original code EXACTLY (whitespace matters)
- Include enough context lines to make the match unique
- You can have multiple SEARCH/REPLACE blocks in one response
- Explain your changes before or after the blocks
#+end_example

** Edit Request Wrapper
=ogent-edit-wrap-prompt= wraps the user instruction with buffer context:

#+begin_example
The user wants to modify code in `{filename}` ({major-mode}).

{user-prompt}

Current code:
```{language}
{source-content}
```

Provide your changes using SEARCH/REPLACE blocks.
#+end_example

* Parser and Validation

Implemented in =lisp/ogent-edit-format.el= (data structure, prompts) and
=lisp/ogent-edit-parse.el= (parsing, validation, filtering).

** Data Structure
#+begin_src elisp
(cl-defstruct ogent-edit
  "A single code edit extracted from LLM response."
  id               ; unique identifier (ogent-edit-001, etc.)
  old-text         ; exact text to search for
  new-text         ; replacement text
  source-buffer    ; buffer containing source code
  source-file      ; file path (if file-backed)
  start-pos        ; position where old-text was found (after validation)
  end-pos          ; end position of old-text
  source-marker    ; marker in source buffer for navigation
  companion-marker ; marker in companion buffer for navigation
  status           ; pending | applied | rejected | error
  error-message    ; why validation/application failed
  timestamp)       ; when edit was created
#+end_src

** Parsing
=ogent-edit-parse-response= scans a response string for SEARCH/REPLACE
blocks and returns a list of =ogent-edit= structs with =pending= status.
IDs are session-unique (=ogent-edit--generate-id=).

** Validation
=ogent-edit-validate= (and =ogent-edit-validate-all= for lists) checks that
=old-text= exists /uniquely/ in the source buffer:
- 0 matches → status =error=, "SEARCH text not found"
- 1 match → =start-pos= / =end-pos= recorded
- 2+ matches → status =error=, ambiguous match

=ogent-edit-filter-valid= and =ogent-edit-filter-errors= partition the
results for display and error reporting.

* Display Methods

The display pipeline lives in =lisp/ogent-edit-display.el=. The user-facing
selector is:

#+begin_src elisp
(defcustom ogent-edit-display-method 'smerge
  "Method for displaying proposed edits.
Options:
- `smerge': Traditional conflict markers (built-in, always available)
- `overlay': gptel-rewrite style overlays with inline preview and dispatch menu
- `inline-diff': Word-level inline highlighting (requires inline-diff.el)"
  ...)
#+end_src

- =ogent-edit-display= / =ogent-edit-display-all= dispatch a single edit or
  a batch to the configured method; unknown values fall back to smerge.
- =ogent-edit-toggle-display-method= (autoloaded) cycles through
  =ogent-edit-display-methods= (=smerge= → =overlay= → =inline-diff=),
  skipping =inline-diff= when the optional package is not loaded
  (=ogent-edit-inline-diff-available-p=). It is bound to ~t~ in both the
  preview-mode and overlay keymaps so the method can be flipped mid-review.

** Re-anchoring Before Apply
Between validation and application the source buffer may change (streaming
responses, concurrent edits, the user typing). Every apply path therefore
re-anchors the edit via =ogent-edit--re-anchor= before mutating the buffer:
- If the cached =start-pos= / =end-pos= region still holds =old-text=
  exactly, positions are kept.
- Otherwise the buffer is rescanned for =old-text=: a unique match
  re-anchors the edit (with a message); zero or multiple matches set status
  =error= and signal =user-error= *without modifying the buffer*.

This applies to the smerge path (=ogent-edit-apply-as-smerge=), the overlay
path (=ogent-edit-apply-as-overlay=), and the inline-diff path
(=ogent-edit-apply-as-inline-diff=), so a stale edit can never clobber
unrelated text regardless of display method.

** Method: smerge (default)
=ogent-edit-apply-as-smerge= replaces the matched region with conflict
markers (upper = original, lower = AI proposal):

#+begin_example
<<<<<<< original
[old text]
=======
[new text]
>>>>>>> ogent
#+end_example

It enables =smerge-mode= if needed, records a source marker for
navigation, sets status =applied=, and jumps to the conflict.
=ogent-edit-apply-all-as-smerge= applies a batch.

Resolution commands (real bindings on Emacs 29/30):

| Action                 | Command                    | Keybinding             |
|------------------------+----------------------------+------------------------|
| Accept AI change       | =smerge-keep-lower=        | =C-c ^ l=              |
| Reject (keep original) | =smerge-keep-upper=        | =C-c ^ u=              |
| Next conflict          | =smerge-next=              | =C-c ^ n=              |
| Previous conflict      | =smerge-prev=              | =C-c ^ p=              |
| Accept current (ogent) | =ogent-edit-accept-current= | =a= in =ogent-edit-menu= |
| Reject current (ogent) | =ogent-edit-reject-current= | =r= in =ogent-edit-menu= |
| Accept all             | =ogent-edit-accept-all=    | =A= in =ogent-edit-menu= |
| Reject all             | =ogent-edit-reject-all=    | =R= in =ogent-edit-menu= |

The ogent commands resolve the conflict *and* update edit status, then run
=ogent-edit-resolved-hook= so companion logging records the outcome.

/When appropriate:/ always available, works in any Emacs, best when you
want the familiar merge-conflict workflow or need to hand-edit the result
before accepting.

** Method: overlay (gptel-rewrite style)
=ogent-edit-apply-as-overlay= keeps the buffer text untouched and overlays
the region: the proposed text is shown via a =display= property
(=ogent-edit-overlay-new-face=), with an action-hint banner and a dedicated
keymap (=ogent-edit-overlay-map=) active while point is on the overlay:

| Key       | Command                             | Action                        |
|-----------+-------------------------------------+-------------------------------|
| ~RET~ / mouse-1 | =ogent-edit-overlay-dispatch=  | Prompt for an action          |
| ~a~       | =ogent-edit-overlay-accept=         | Accept (replace old with new) |
| ~r~ / ~k~ | =ogent-edit-overlay-reject=         | Reject (drop overlay)         |
| ~d~       | =ogent-edit-overlay-diff=           | Open unified-diff preview     |
| ~e~       | =ogent-edit-overlay-ediff=          | Compare in Ediff              |
| ~m~       | =ogent-edit-overlay-merge=          | Convert to smerge conflict    |
| ~n~ / ~p~ | =ogent-edit-overlay-next= / =ogent-edit-overlay-previous= | Navigate overlays |
| ~t~       | =ogent-edit-toggle-display-method=  | Cycle display method          |

Batch operations: =ogent-edit-apply-all-as-overlay=,
=ogent-edit-overlay-accept-all=, =ogent-edit-overlay-reject-all=.

=ogent-edit-overlay-default-action= (defcustom, default nil) can
auto-trigger =accept=, =merge=, =diff=, =ediff=, or =dispatch= as soon as
an overlay is created; nil waits for explicit user action. Accepting an
overlay re-anchors the edit before splicing in the replacement text, so
the apply is safe even if the buffer moved underneath the overlay.

/When appropriate:/ the least intrusive review — the buffer content is not
rewritten until you accept, and each edit carries its own single-key
actions. Best for interactive review of a handful of edits.

** Method: inline-diff (optional package)
=ogent-edit-apply-as-inline-diff= requires the optional =inline-diff.el=
package (checked via =ogent-edit-inline-diff-available-p=; a clear
=user-error= is signaled when missing). It re-anchors, replaces the old
text with the new text, then calls =inline-diff-words-region= to paint
word-level change highlights over the replacement. Accepting or rejecting
goes through inline-diff's own commands; ogent installs buffer-local
functions on =inline-diff-accept-hook= / =inline-diff-reject-hook= so all
pending edits are marked accepted/rejected and =ogent-edit-resolved-hook=
fires. Status stays =pending= until the user resolves.
=ogent-edit-apply-all-as-inline-diff= applies a batch (later edits are
appended rather than clearing earlier highlights).

/When appropriate:/ the most granular view — word-level rather than
block-level — for users who have =inline-diff.el= installed and want to
see precisely what changed inside dense lines.

* Diff Preview and Review Buffers

** Single-edit Diff Preview (=*ogent-diff*=)
=ogent-edit-preview-diff= (autoloaded) renders one edit as a unified diff
(=ogent-edit--generate-unified-diff=) in a bottom side window
(height =ogent-edit-preview-window-height=, default 0.3), with =diff-mode=
highlighting plus =ogent-edit-preview-mode= for review keys:

| Key         | Command                            | Action                          |
|-------------+------------------------------------+---------------------------------|
| ~a~ / ~RET~ | =ogent-edit-preview-accept=        | Apply edit (as smerge conflict) |
| ~r~ / ~q~   | =ogent-edit-preview-reject=        | Reject and close preview        |
| ~n~ / ~p~   | =diff-hunk-next= / =diff-hunk-prev= | Navigate hunks                 |
| ~t~         | =ogent-edit-toggle-display-method= | Cycle display method            |

Accepting applies the edit via =ogent-edit-apply-as-smerge= (re-anchoring
included; a stale/ambiguous edit keeps the preview open and reports why)
and tracks it in the source buffer's =ogent-edit--pending-edits= so
resolution is logged like any other edit. Rejecting sets status
=rejected=, untracks, and runs =ogent-edit-resolved-hook=. This preview is
also reachable from an overlay via ~d~.

** Multi-edit Review Buffer (=*ogent-edits*=)
=lisp/ogent-edit-diff.el= provides a Magit-style review buffer for a whole
batch of proposals: =ogent-edit-show-diff-buffer= (autoloaded; ~D~ in
=ogent-edit-menu=) calls =ogent-edit-diff-show= on the pending edits.
Features:
- Collapsible file and hunk sections (soft dependency on =magit-section=;
  falls back to a basic plain-text rendering when it is not installed)
- Stage/unstage semantics for partial acceptance
- Batch accept/reject

=ogent-edit-diff-mode= keymap:

| Key         | Command                            | Action                       |
|-------------+------------------------------------+------------------------------|
| ~n~ / ~p~   | =ogent-edit-diff-next-hunk= / =ogent-edit-diff-prev-hunk= | Navigate hunks |
| ~M-n~ / ~M-p~ | =ogent-edit-diff-next-file= / =ogent-edit-diff-prev-file= | Navigate files |
| ~TAB~       | =ogent-edit-diff-toggle-section=   | Collapse/expand section      |
| ~s~ / ~u~   | =ogent-edit-diff-stage= / =ogent-edit-diff-unstage= | Stage/unstage hunk |
| ~S~ / ~U~   | =ogent-edit-diff-stage-all= / =ogent-edit-diff-unstage-all= | Stage/unstage all |
| ~a~         | =ogent-edit-diff-accept-at-point=  | Accept hunk at point         |
| ~r~         | =ogent-edit-diff-reject-at-point=  | Reject hunk at point         |
| ~A~         | =ogent-edit-diff-accept-staged=    | Accept all staged hunks      |
| ~R~         | =ogent-edit-diff-reject-all=       | Reject everything            |
| ~RET~       | =ogent-edit-diff-goto-source=      | Jump to source location      |
| ~g~         | =ogent-edit-diff-refresh=          | Re-render                    |
| ~?~         | =ogent-edit-diff-help=             | Key summary                  |
| ~q~         | =quit-window=                      | Close                        |

Customs: =ogent-edit-diff-buffer-name= (default =*ogent-edits*=),
=ogent-edit-diff-window-height= (default 0.4). Under Evil/Doom the mode
map is registered as an overriding map (see =ogent-evil-setup-display-mode=
in =lisp/ogent-keys.el=) so the single-key affordances fire.

* Request Flows

All request commands live in =lisp/ogent-edit.el= and funnel through a
common pipeline.

** Common Pipeline
=ogent-request-edit= (autoloaded) is the shared entry point:
1. Requires gptel (=ogent-edit--ensure-gptel=) and a file-backed buffer.
2. Resolves the edit target: explicit =START= / =END= bounds from callers,
   else the active region, else the whole buffer.
3. Wraps the instruction with =ogent-edit-wrap-prompt= and resolves the
   edit model/backend (=ogent-models-effective-model=,
   =ogent-gptel-resolve-backend=).
4. Sends via =gptel-request= with =ogent-edit-system-prompt=, streaming
   when the model supports it; request context is stashed in
   =ogent-edit--pending-request=. Failures can offer provider login
   (=ogent-provider-maybe-offer-login=).
5. The callback hands the full response to =ogent-edit--process-response=:
   parse → validate → log proposals/errors to the companion (when
   =ogent-edit-log-to-companion=) → then either
   - auto-apply valid edits (=ogent-edit-auto-apply=, off by default;
     undo via standard Emacs undo), or
   - display them with the configured method (=ogent-edit-auto-display=,
     on by default), track them in =ogent-edit--pending-edits=, and jump
     to the first edit.

** Quick Edit
=ogent-quick-edit= (~C-c . k~) prompts for a one-line instruction and
picks a tight target automatically: active region → enclosing defun →
current line (=ogent-edit--quick-target=); a prefix argument widens to the
whole buffer. The instruction is wrapped with a scope hint
(=ogent-edit--quick-prompt=) and dispatched through =ogent-request-edit=.
Use it for "rename this", "add a docstring here" class changes without
composing a full request.

** Diagnostic Repair
Two commands turn Flymake/Flycheck diagnostics into repair requests:
- =ogent-fix-diagnostic= (~C-c . f~): takes the diagnostic at point
  (Flymake preferred, Flycheck fallback), targets the diagnostic's span
  widened to the enclosing defun/line (prefix = whole buffer), and builds
  a repair prompt from the diagnostic text
  (=ogent-edit--diagnostic-prompt=).
- =ogent-fix-buffer-diagnostics= (~C-c . F~): collects, dedupes, and
  severity-sorts every diagnostic in the buffer and requests one
  whole-buffer edit fixing all of them
  (=ogent-edit--buffer-diagnostics-prompt=); prefix prompts for extra
  guidance.

** AI Speed Edit
=ogent-ai-speed-edit= (~C-c . v~) is the "just fix it" command: it gathers
all buffer diagnostics, chooses the most relevant target around point
(=ogent-edit--ai-speed-target=), and sends a prompt containing the target
code, the diagnostics overlapping the target, and the remaining buffer
diagnostics as context (=ogent-edit--ai-speed-prompt=), letting the model
decide the best small edit. Prefix widens to the whole buffer.

** Zen Rewrite Path (related)
Org Zen transcripts have their own inline-edit subsystem
(=lisp/ogent-zen-edit.el=): =ogent-zen-edit-dwim= (~C-c . C-e~) requests a
rewrite through the Zen transcript, and =ogent-zen-apply-last-edit=
(~C-c . C-a~) parses the SEARCH/REPLACE response and previews it with
=inline-diff= directly. It shares the response format but not the
=ogent-edit= display pipeline above.

* Companion Buffer Logging

Implemented in =lisp/ogent-edit-log.el=; enabled by
=ogent-edit-log-to-companion= (default t).

** Log Entry Format
Each edit operation creates an entry in the companion Org buffer:

#+begin_example
** Edit: ogent-edit-001 [2025-12-09 Mon 14:30]
:PROPERTIES:
:OGENT_EDIT_ID: ogent-edit-001
:SOURCE_FILE: /path/to/file.el
:STATUS: pending
:END:

*** Proposed Change
#+begin_src diff
- (defun foo ()
-   "Old docstring."
-   (old-implementation))
+ (defun foo ()
+   "New improved docstring."
+   (new-implementation))
#+end_src

*** Resolution [2025-12-09 Mon 14:32]
Status: accepted
#+end_example

** Logging Functions
- =ogent-edit-log-proposal= / =ogent-edit-log-all-proposals=: create the
  Org heading with properties drawer and diff block, storing a
  companion marker on the edit struct.
- =ogent-edit-log-resolution=: locates the entry via
  =ogent-edit--find-log-marker=, updates the =STATUS= property, and adds a
  resolution timestamp.
- =ogent-edit-log-errors=: records validation failures.

* Marker Synchronization and Resolution Tracking

** Markers
Rather than a separate marker struct, each =ogent-edit= carries
=source-marker= (set when the edit is displayed) and =companion-marker=
(set when the proposal is logged). Displayed edits are tracked in the
buffer-local =ogent-edit--pending-edits= list.

** Navigation Commands
- =ogent-edit-goto-source= (~C-c . s~): from a companion log entry
  (identified by the =OGENT_EDIT_ID= property), jump to the edit's
  location in the source buffer.
- =ogent-edit-goto-companion= (~C-c . C~): from a source-buffer edit at
  point, jump to its companion log entry.

** Resolution Hook
=ogent-edit-resolved-hook= runs whenever an edit is accepted or rejected —
from smerge resolution (=ogent-edit--smerge-resolved-hook=,
=ogent-edit-accept-current= / =ogent-edit-reject-current=), overlay
actions, inline-diff hooks, the diff preview, or the review buffer. Each
function receives the =ogent-edit= struct; companion logging subscribes
here.

* Transient Menu

=ogent-edit-menu= (~C-c . e~) is display-method aware: its header shows
the pending count and current method, and its resolution commands dispatch
to the right backend (smerge commands vs overlay commands) via dispatch
helpers (=ogent-edit--accept-current-dispatch=,
=ogent-edit--reject-current-dispatch=, =ogent-edit--next-dispatch=,
=ogent-edit--prev-dispatch=, =ogent-edit--accept-all-dispatch=,
=ogent-edit--reject-all-dispatch=).

| Group        | Key | Action                                               |
|--------------+-----+------------------------------------------------------|
| Current Edit | ~a~ | Accept (dispatches per display method)               |
| Current Edit | ~r~ | Reject (dispatches per display method)               |
| Current Edit | ~n~ / ~p~ | Next / previous edit (transient)               |
| All Edits    | ~A~ / ~R~ | Accept all / reject all                        |
| Request      | ~v~ | =ogent-ai-speed-edit=                                |
| Request      | ~f~ | =ogent-fix-diagnostic=                               |
| Request      | ~F~ | =ogent-fix-buffer-diagnostics=                       |
| Request      | ~k~ | =ogent-quick-edit=                                   |
| Request      | ~e~ | =ogent-request-edit=                                 |

| Request      | ~q~ | Quit                                                 |
| Overlay      | ~d~ / ~E~ / ~m~ | Diff / Ediff / Merge (only in overlay mode) |

* Testing Strategy

** Parser Tests
- Valid single edit block
- Multiple edit blocks
- Edit with surrounding prose
- Malformed blocks (missing markers)
- Empty SEARCH or REPLACE sections
- Unicode content

** Validation Tests
- OLD text found once (success)
- OLD text not found (error)
- OLD text found multiple times (error)
- Whitespace sensitivity

** Display Tests
- Each display method applies and resolves correctly
- Re-anchoring: stale positions recovered on unique match, refused on
  missing/ambiguous text
- Preview and review buffers accept/reject/stage as advertised

** Integration Tests
- Full request → parse → display flow (per method)
- Accept/reject updates status and runs =ogent-edit-resolved-hook=
- Companion logging records all actions
- Marker navigation works both directions
